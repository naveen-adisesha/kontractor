1. The Problem: Why Not Webhooks?

Mutating Admission Webhooks Have Real Costs

A MutatingAdmissionWebhook is the Kubernetes-native way to modify resources before they're persisted. But deploying one means:

  • Running a webhook server inside the cluster (Deployment + Service)
  • Managing TLS certificates for the webhook endpoint
  • Creating a MutatingWebhookConfiguration CRD
  • Handling failure modes (if the webhook is down, pod creation blocks or bypasses)
  • Cluster admin access to install the webhook
  • Potential latency on every pod creation in the cluster

For a concept like Kontractor — where we want to inject infrastructure requirements from a contract into Helm-rendered manifests — a webhook is overkill for many use cases. The mutation can happen before the YAML ever reaches the cluster.

2. The Post-Renderer Approach

Helm's --post-renderer: The Sweet Spot

Helm natively supports post-renderers — external programs that receive rendered YAML on stdin and output modified YAML on stdout. This is a first-class Helm feature, not a hack.

helm template / install → Rendered YAML (stdin) → kontractor-post-render → Mutated YAML (stdout) → kubectl apply

The post-renderer:

  1. Reads the Container Contract from a specified file
  2. Evaluates feature flags (e.g., --features=tls)
  3. Resolves variable substitutions (e.g., ${RELEASE} → actual release name)
  4. Parses all YAML documents from stdin
  5. For each workload (Deployment, StatefulSet, DaemonSet, Job, CronJob):
    • Injects missing environment variables
    • Adds missing volumes to the pod spec
    • Adds missing volume mounts to the main container
    • Annotates the resource with contract metadata
  6. Outputs all resources (mutated + untouched) on stdout

5. Component Breakdown

5a. Contract Loader (pkg/contract/)

The contract package defines Go structs that mirror the kontractor.io/v1 ContainerContract schema. Key types:

TypePurpose
ContainerContractRoot type: apiVersion, kind, metadata, spec
SpecFeatures, env, mounts, secrets, ports, health, security, rbac
FeatureNamed boolean flag with description and default value
EnvSpecRequired + optional + conditional environment variables
MountSpecRequired + optional + conditional volume mounts
WhenClauseFeature-based conditional: when: { feature: tls }

The loader validates apiVersion and kind on parse, failing early on invalid contracts.

The SubstituteVars function performs ${RELEASE}-style variable resolution across all string fields in the contract before mutation begins.

5b. YAML Parser (pkg/parser/)

Uses gopkg.in/yaml.v3's AST node representation rather than deserializing into typed structs. This preserves:

  • Original YAML ordering and formatting
  • Comments (where possible)
  • Unknown fields the mutator doesn't care about

Key functions:

FunctionPurpose
ParseManifests()Splits multi-document YAML on ---, parses each into *yaml.Node
GetField()Traverses nested mapping nodes by path (e.g., "spec", "template", "spec")
GetStringField()Shorthand for getting a scalar value by path
SerializeManifests()Re-serializes all nodes back to multi-document YAML
IsWorkload()Returns true for Deployment, StatefulSet, DaemonSet, Job, CronJob

5c. Mutation Engine (pkg/mutator/)

The core mutation logic. For each workload resource:

// Pseudocode
func MutateWorkload(node, contract, features):
    podSpec  = findPodSpec(node)           // handles CronJob nesting
    container = podSpec.containers[0]

    for env in contract.ResolvedEnv(features):
        if !hasEnvVar(container, env.Name):
            addEnvVar(container, env)       // value or secretKeyRef

    for mount in contract.ResolvedMounts(features):
        volName = volumeNameFromPath(mount.Path)
        if !hasVolume(podSpec, volName):
            addVolume(podSpec, mount)       // secret/configMap/emptyDir
        if !hasVolumeMount(container, mount.Path):
            addVolumeMount(container, mount)

    addContractAnnotation(node, contract, features)

Idempotency: The mutator checks for existing env vars (by name), volumes (by name), and mounts (by mountPath) before injecting. Running the post-renderer twice produces identical output.

secretsecret.secretNamesecretName field
configmapconfigMap.nameconfigMapName field
ephemeral / emptyDiremptyDir.sizeLimitmaxSize field

8. Conditional Feature System

How Features Work

The contract defines a features map with named boolean flags. Each flag has a description and a default value:

spec:
  features:
    tls:
      description: "Enable HTTPS with TLS certificates"
      default: false

Requirements can then be made conditional using when clauses:

env:
  required:            # Always injected
    - name: ECHO_TEXT
      default: "Hello!"
  conditional:          # Only when feature is enabled
    - when: { feature: tls }
      env:
        - name: TLS_ENABLED
          value: "true"
Feature Resolution Flow:

  CLI flags: --features=tls        Contract defaults:
  Env var: KONTRACTOR_FEATURES       tls: false
       │                               │
       ▼                               ▼
  ┌────────────────────────────────────────┐
  │         Feature Resolver               │
  │                                        │
  │  For each feature in contract:         │
  │    if flag provided → use flag value   │
  │    else → use contract default         │
  │                                        │
  │  Result: { tls: true }                 │
  └───────────────┬────────────────────────┘
                  │
     ┌────────────┴────────────┐
     ▼                         ▼
  ┌─────────┐           ┌──────────────┐
  │ Required│           │ Conditional  │
  │ always  │           │ when: tls    │
  │ inject  │           │ → inject     │
  └─────────┘           └──────────────┘

9. Integration Methods

Method 1: Direct Pipeline

Render with helm template, pipe through the binary, apply with kubectl:

helm template myrelease ./mychart | \
  ./bin/kontractor-post-render \
    --contract=contract.yaml \
    --features=tls \
    --set-vars=RELEASE=myrelease | \
  kubectl apply -f -

Method 2: Helm --post-renderer Flag

Use the wrapper script with Helm's native post-renderer support:

export KONTRACTOR_CONTRACT=contract.yaml
export KONTRACTOR_FEATURES=tls

helm install myrelease ./mychart \
  --post-renderer ./kontractor-post-render.sh

This works with helm install, helm upgrade, and helm template.

Method 3: Helm Plugin

Install as a Helm plugin for global availability:

helm plugin install ./kontractor-helm-plugin

# Then use anywhere
helm kontractor install myrelease ./mychart \
  --contract=contract.yaml --features=tls

Method 4: CI/CD Pipeline

Integrate into GitOps workflows:

# In a CI pipeline (GitHub Actions, GitLab CI, etc.)
helm template myrelease ./mychart \
  | kontractor-post-render \
      --contract=contract.yaml \
      --features="${FEATURES}" \
      --set-vars="RELEASE=myrelease" \
  > rendered-manifests.yaml

# Validate, review, then apply
kubectl apply -f rendered-manifests.yaml

10. Post-Renderer vs Webhook

AspectPost-Renderer CurrentMutating Webhook Future
Cluster infrastructureNone requiredWebhook server + TLS + Service
InstallationSingle binaryHelm chart + CRD + RBAC
When mutation happensBefore apply (render time)At admission (runtime)
VisibilitySee mutated YAML before applyMutations happen opaquely
Failure modePipeline stops; no partial applyPod creation blocked or bypassed
ScopeHelm releases onlyAll pod creation (kubectl, operators, etc.)
Auditinggit diff on rendered YAMLAudit logs + annotations
Dry-runBuilt-in --dry-runRequires separate tooling
Air-gapped supportWorks offlineRequires in-cluster connectivity
GitOps compatibleYes (pre-commit rendering)Post-apply only

Recommended Strategy: Start with Post-Renderer, Graduate to Webhook

The post-renderer is the right choice for most teams getting started with Kontractor. It requires zero cluster-side infrastructure, integrates with existing CI/CD, and provides full visibility into mutations before they're applied.

Once contract-driven deployment is proven, the same contract schema can drive a webhook for universal coverage (non-Helm workloads, operator-created pods, etc.).
