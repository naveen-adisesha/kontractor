Kontractor

Declarative Container Infrastructure Requirements — An open standard for self-describing container images

1 The Problem

Containers Are Opaque

A container image is a black box. It needs environment variables, TLS certificates, persistent storage, configuration files, secrets, and RBAC permissions. But there is no standardized way for the container to declare those requirements.

Instead, platform engineers manually author Helm values, Kustomize overlays, or raw YAML — often by reading READMEs, reverse-engineering Dockerfiles, or relying on tribal knowledge.

Configuration Drift Is Silent

When a container image is upgraded, the deployment config may not match. A new env var, a changed mount path, or an additional TLS certificate silently breaks the deployment.

The image and its deployment configuration are independently versioned artifacts with no enforced coupling. The container crashes at runtime instead of failing at deploy time.

2 The Solution: Container Contracts

What if containers could tell Kubernetes what they need?

A Container Contract is a machine-readable YAML manifest embedded in or attached to a container image. It declares everything the container requires from the platform: environment variables, volume mounts, secrets, configmaps, ports, health checks, RBAC permissions, and security context.

A Kubernetes admission webhook reads the contract at deploy time and either validates that requirements are met or provisions missing resources automatically.

Example Contract

apiVersion: kontractor.io/v1
kind: ContainerContract
metadata:
  name: echo-server
  version: "1.0.0"
  image: myregistry/echo-server

spec:
  features:
    tls:
      description: "Enable HTTPS"
      default: false

  env:
    required:
      - name: APP_PORT
        default: "8080"
    conditional:
      - when: { feature: tls }
        env:
          - name: TLS_CERT_PATH
            default: "/etc/tls/tls.crt"
          - name: TLS_KEY_PATH
            default: "/etc/tls/tls.key"

  mounts:
    required:
      - path: /tmp
        type: ephemeral
        maxSize: 64Mi
    conditional:
      - when: { feature: tls }
        mounts:
          - path: /etc/tls
            type: secret
            secretName: "${RELEASE}-tls"

  ports:
    - name: http
      containerPort: 8080
      protocol: TCP

  health:
    readiness:
      httpGet:
        path: /healthz
        port: 8080

  security:
    runAsNonRoot: true
    readOnlyRootFilesystem: true
    dropCapabilities: ["ALL"]

  rbac:
    rules:
      - apiGroups: [""]
        resources: ["configmaps", "secrets"]
        verbs: ["get", "list", "watch"]

3 Key Concepts

3.1 Feature Flags & Conditional Requirements

Not every requirement is always active. TLS mounts should only exist when TLS is enabled. Backup volumes only matter when backup is configured. Contracts handle this with feature flags:

spec:
  features:
    tls:
      description: "SSL/TLS encryption"
      default: false
    ha:
      description: "High availability mode"
      default: false

  mounts:
    required:
      - path: /tmp              # Always needed
        type: ephemeral
    conditional:
      - when: { feature: tls }  # Only when TLS enabled
        mounts:
          - path: /etc/tls
            type: secret
      - when:                    # Compound: TLS AND HA
          all:
            - { feature: tls }
            - { feature: ha }
        mounts:
          - path: /etc/tls/replicated
            type: persistent

Operators toggle features via Helm values or pod annotations. The webhook evaluates the conditions and includes/excludes requirements accordingly.

3.2 Condition Expression Types

ExpressionSyntaxUse Case
Single featurewhen: { feature: tls }TLS volumes only when TLS enabled
All (AND)when: { all: [{ feature: tls }, { feature: mtls }] }mTLS-specific config
Any (OR)when: { any: [{ feature: tls }, { feature: istio }] }Cert mount for either mode
Negationwhen: { not: { feature: tls } }Plaintext-only config
Env checkwhen: { env: { name: MODE, equals: "debug" } }Debug-only volumes
Always(no when clause)Unconditional requirements

Webhook Modes

Audit Mode

Logs warnings for unmet requirements but allows the pod to start. Zero risk. Ideal for rollout.

Safe

Validate Mode

Rejects pods that don't meet required contract requirements. Clear error messages.

Recommended

Mutate Mode

Automatically injects missing volumes, env vars, and mounts from the contract.

Advanced

5 Benefits

Eliminate Configuration Drift

The contract is baked into the image. When a developer adds a new env var, they update the contract in the same PR. The next deployment automatically validates it.

Impact: Critical

Deploy-Time Validation

"Pod rejected: missing required secret 'app-tls' with keys [tls.crt, tls.key]" instead of a cryptic crash 10 minutes after deployment.

Impact: Critical

Self-Documenting Images

Run kontractor inspect myimage:v2 and see every env var, mount, port, and secret the container needs. No README hunting.

Impact: High

Simplified Helm Charts

Charts only need to declare operator-specific overrides (replicas, resources, feature flags). All container-intrinsic requirements come from the contract.

Impact: High

Automated RBAC

The contract declares exactly which RBAC rules the container needs. The webhook or Helm chart auto-generates Role and RoleBinding resources. No more guessing.

Impact: Medium

Cross-Platform Portability

The same contract works on vanilla K8s, OpenShift, EKS, GKE, or any platform running the webhook. Container requirements are platform-agnostic.

Impact: Medium

7 Challenges & Mitigations

Security: Untrusted Contracts

A malicious container could request privileged mounts or excessive RBAC. Mitigation: Contracts request, policies permit. Pair Kontractor with Kyverno/OPA to constrain what contracts can ask for.

Image Pull Latency

Reading embedded contracts requires pulling image layers. Mitigation: Use OCI Referrer artifacts (no pull needed) or cache contracts in the webhook.

Migration Complexity

Existing images don't have contracts. Mitigation: Start with audit mode. Provide a kontractor generate CLI that scans Dockerfiles and existing Helm values to bootstrap contracts.

Webhook Availability

If the webhook is down, pods can't start. Mitigation: HA deployment (3+ replicas), failurePolicy: Ignore with alerting, contract data as Helm fallback.

8 Getting Started

Try the Sample Echo Server

# Clone the repo
git clone https://github.com/YOUR_USERNAME/kontractor.git
cd kontractor

# Deploy without TLS
helm install echo-server charts/echo-server \
  -n kontractor --create-namespace

# Deploy with TLS enabled
kubectl create secret tls echo-tls \
  --cert=tls.crt --key=tls.key -n kontractor

helm install echo-tls charts/echo-server \
  -n kontractor \
  --set contract.features.tls=true \
  --set tls.secretName=echo-tls

# Inspect the contract
kontractor-validate examples/echo-server.contract.yaml

Why This Matters

Dockerfiles introduced EXPOSE, VOLUME, and ENV as the first attempt at self-declaring container requirements. Kubernetes ignored them. The Container Contract pattern finishes what Dockerfiles started — with the rigor, validation, and automation to make it work at production scale.

Containers shouldn't be opaque. They should tell the platform what they need, and the platform should listen.
