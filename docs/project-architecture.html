3 File-by-File Deep Dive

3.1 spec/contract-schema.yaml — The Standard

This is the heart of the project. It defines the kontractor.io/v1 ContainerContract schema using JSON Schema (2020-12 draft). Every other file either implements this schema or consumes it.

Schema Structure
ContainerContract
├── apiVersion: "kontractor.io/v1"
├── kind: "ContainerContract"
├── metadata
│   ├── name          ← unique contract identifier
│   ├── version       ← tracks image version
│   ├── image         ← container image repo
│   └── description
└── spec
    ├── features       ← named boolean flags (tls, ha, backup...)
    │   └── {name}: { description, default }
    ├── env            ← environment variable requirements
    │   ├── required[]    ← must always be set
    │   ├── optional[]    ← defaults provided
    │   └── conditional[] ← gated by features
    │       └── { when: {feature: X}, env: [...] }
    ├── mounts         ← volume mount requirements
    │   ├── required[]    ← always needed (e.g. /tmp)
    │   ├── optional[]
    │   └── conditional[] ← e.g. TLS secret mount
    ├── secrets        ← K8s secrets needed
    ├── configMaps     ← K8s configmaps needed
    ├── ports          ← container ports exposed
    ├── health         ← startup/readiness/liveness probes
    ├── security       ← security context requirements
    ├── rbac           ← RBAC rules needed (namespace + cluster)
    └── initContainers ← init container specifications

Key Schema Features

FeatureSchema TypePurpose
whenClauseoneOf (feature / all / any / not / env)Conditional expressions for gating requirements
mount.typeenumephemeral, persistent, secret, configMap, ephemeral-subpath
probe.whenconditionalOverrideOverride probe settings when features are enabled
envVar.secretRefobjectReference env value from K8s secret

3.4 cmd/kontractor-validate/main.go — CLI Validator

A Go CLI that parses a contract YAML file and displays its contents in a structured format. This is the first step toward full validation.

CLI Data Flow
.contract.yamlon disk or in image
→
YAML Parsergopkg.in/yaml.v3
→
Go StructsContainerContract
Spec, Env, Mount...
→
Formatted OutputFeatures, Env, Mounts
Ports, RBAC, Security

Go Type Hierarchy

ContainerContract
├── Metadata { Name, Version, Image, Description }
└── Spec
    ├── Features        map[string]Feature { Description, Default }
    ├── Env             EnvSpec
    │   ├── Required    []EnvVar { Name, Description, Default, Value }
    │   ├── Optional    []EnvVar
    │   └── Conditional []ConditionalEnvGroup { When, Env }
    ├── Mounts          MountSpec
    │   ├── Required    []Mount { Path, Type, MaxSize, SecretName }
    │   └── Conditional []ConditionalMountGroup { When, Mounts }
    ├── Ports           []Port { Name, ContainerPort, Protocol }
    ├── Health          HealthSpec { Startup, Readiness, Liveness *Probe }
    ├── Security        SecuritySpec { RunAsNonRoot, ReadOnly, UID, Caps }
    └── RBAC            RBACSpec { Rules, ClusterRules []RBACRule }

6 How Conditional TLS Works in Detail

Conditional Flow Through the System
Operator setscontract.features.tls: true
in values.yaml or --set
→
deployment.yaml{{- if .Values.contract.features.tls }}
evaluates to true
→
env:
  - TLS_ENABLED: "true"
  - TLS_CERT_PATH: /etc/tls/tls.crt
  - TLS_KEY_PATH: /etc/tls/tls.key
volumeMounts:
  - /etc/tls (readOnly)
volumes:
  - secret: echo-server-tls

The contract declares what TLS needs. The template implements the conditional. The webhook (future) will validate and auto-inject.

In the Contract (what the container declares)

conditional:
  - when: { feature: tls }
    mounts:
      - path: /etc/tls
        type: secret
        secretName: "${RELEASE}-tls"
        keys: ["tls.crt", "tls.key"]
        readOnly: true

In the Template (how Helm renders it)

{{- if .Values.contract.features.tls }}
- name: tls-certs
  secret:
    secretName: {{ .Values.tls.secretName |
      default (printf "%s-tls"
        (include "echo-server.fullname" .)) }}
{{- end }}
